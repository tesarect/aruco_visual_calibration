# aruco_moveit_config

`aruco_moveit_config` is a MoveIt2 configuration package for the UR3e arm
with an RG2 parallel gripper, generated by the MoveIt Setup Assistant from
`ur_description`'s `ur.urdf.xacro` — the same URDF source that matches the
robot actually spawned in the Gazebo simulation. It is pure configuration
(SRDF, YAML, xacro): there is no node source code here, only the semantic
robot description and planning parameters that `move_group` and downstream
nodes like `trajectory_planner` load at startup.

Most of what's in this package is stock Setup Assistant output for a UR3e +
parallel-gripper robot. This page focuses on the one deliberate,
project-specific change, then briefly notes how each remaining config file
compares to what the Setup Assistant would generate by default.

## The `tip_link` change: `tool0` → `rg2_gripper_aruco_link`

In `config/name.srdf`, the `ur_manipulator` planning group is defined as a
kinematic chain from `base_link` to a `tip_link`:

```xml
<group name="ur_manipulator">
    <chain base_link="base_link" tip_link="rg2_gripper_aruco_link"/>
</group>
```

In a stock UR MoveIt configuration, `tip_link` is `tool0` — the flange /
tool-mounting frame at the end of the arm's own kinematic chain, before any
end-effector-specific hardware is attached. This project's SRDF instead sets
`tip_link` to `rg2_gripper_aruco_link`, the frame the ArUco marker is
rigidly mounted at on the RG2 gripper.

This is a relabeling, not a mechanical change: `rg2_gripper_aruco_joint`,
the joint connecting the gripper structure to the marker frame, is a fixed
joint — a rigid mount with no actuation. Extending the chain's tip past
`tool0` and the gripper body out to `rg2_gripper_aruco_link` therefore adds
zero real degrees of freedom to the `ur_manipulator` group; it only changes
which frame MoveIt treats as the group's end when planning and solving IK.

The reason this matters for this project specifically: it lets calling code
target the ArUco marker's own frame directly. `trajectory_planner` (see
[visual_calibration_moveit.md](./visual_calibration_moveit.md)) calls
`move_group_interface_.setEndEffectorLink(config.end_effector_frame)` before
every planned move, with `end_effector_frame` set to
`rg2_gripper_aruco_link` in `trajectory_planner_sim.yaml`. Because the SRDF
already defines the `ur_manipulator` chain's tip at that link, this
`setEndEffectorLink()` call combined with `setPoseTarget()` is enough to
plan directly to a desired marker-frame pose. Without the `tip_link` change,
the same goal would require computing an equivalent `tool0` pose from the
desired `rg2_gripper_aruco_link` pose (by composing in the fixed
`tool0 → rg2_gripper_aruco_link` offset) before every planning call — extra
math duplicated at every call site instead of being expressed once, in
configuration.

The SRDF comment documenting this change also states its own rollback path:
if targeting `rg2_gripper_aruco_link` as the chain tip causes planning or IK
problems, `tip_link` can be reverted to `tool0`.

The `gripper` group (`rg2_gripper_base_joint` plus the finger/thumb joints),
its `gripper_open` / `gripper_close` group states, and the `end_effector`
tag pairing the `gripper` group with `ur_manipulator` are standard Setup
Assistant output for a parallel-gripper robot and are not part of this
project-specific change.

## Remaining config files

- **`config/kinematics.yaml`** — stock Setup Assistant output: the default
  `KDLKinematicsPlugin` for the `ur_manipulator` group, with default search
  resolution and timeout.
- **`config/joint_limits.yaml`** — stock Setup Assistant output: velocity
  and acceleration limits pulled from the URDF, with the Assistant's usual
  0.1 default velocity/acceleration scaling factors for beginners.
- **`config/moveit_controllers.yaml`** — stock Setup Assistant output for
  the two controllers this robot's `ros2_control` setup exposes:
  `joint_trajectory_controller` (`FollowJointTrajectory`, six arm joints)
  and `gripper_controller` (`GripperCommand`, the gripper's left finger
  joint).
- **`config/initial_positions.yaml`** — stock Setup Assistant output: all
  joints default to `0`, used to seed the fake `ros2_control` system's
  initial joint state.
- **`config/name.urdf.xacro`** — shows a deliberate change: its
  `initial_positions_file` xacro arg is named
  `ros2_control_initial_positions_file` rather than the Setup Assistant's
  usual `initial_positions_file`. Xacro args share a single global namespace
  per process, and `ur.urdf.xacro` (included by this file) already declares
  its own `initial_positions_file` arg for the arm's `ros2_control` block.
  Using the same name here would have silently fed both consumers the same
  YAML file, even though the arm's block expects a flat joint-name map and
  this package's gripper block expects one wrapped under
  `initial_positions:` — the rename avoids that collision.
- **`config/name.ros2_control.xacro`** — stock Setup Assistant output: a
  `mock_components/GenericSystem` hardware block with position/velocity
  interfaces for the six arm joints plus the gripper's left finger joint,
  seeded from the initial-positions YAML.
- **`config/pilz_cartesian_limits.yaml`** — stock Setup Assistant output:
  default Cartesian velocity/acceleration/deceleration limits for the Pilz
  planner.
